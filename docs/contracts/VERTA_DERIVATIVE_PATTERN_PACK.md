# Verta Derivative Pattern Pack

## Purpose

This document defines the first Verta-derived owner seam admitted into the ATLAS ecosystem as a Playbook-owned derivative surface.

The seam is documentation and contract only. It does not promote raw `Verta-Core` source, and it does not create executable runtime behavior.

## Owner And Boundary

- owner repo: `playbook`
- seam id: `verta-derivative-pattern-pack`
- seam class: `governance-runtime`
- source posture: `visible_untrusted` provenance only
- admitted artifact: reviewed derivative pattern pack

Playbook owns:

- reusable governance interpretation of reviewed Verta-derived principles
- pattern and contract wording for later deterministic reuse
- provenance-preserving references to reviewed derivative notes

Playbook does not own:

- the raw `repos/Verta-Core` checkout
- the raw `repos/Verta-Core.zip` archive
- execution, approval, or privileged operator behavior
- direct promotion of unreviewed historical material into canonical doctrine

## Provenance

This artifact is derivative-only. It exists to convert reviewed Verta-adjacent historical material into Playbook-owned pattern language without importing raw source into owner truth.

Only reviewed derivative notes may feed this seam.

Allowed sources:

- `../../../docs/knowledge/promotions/atlas--verta-historical-playbook-principles-20260417.md`
- `../../../docs/knowledge/promotions/atlas--verta-historical-convergence-intent-20260417.md`
- `../../../docs/knowledge/promotions/atlas--verta-historical-scope-boundaries-20260417.md`
- `../../../docs/knowledge/reviews/verta-core.md`
- `../../../docs/knowledge/reviews/verta-core-scrub-report.md`

Disallowed sources for direct adoption:

- `repos/Verta-Core/**`
- `repos/Verta-Core.zip`
- executable scripts, launchers, eval tooling, or private config extracted from the raw archive

## Trust Boundary

This seam is safe only because it remains non-executable and provenance-bounded.

- Playbook may admit rewritten pattern language, review criteria, and deterministic promotion rules.
- Playbook may not admit raw Verta text, raw source, launchers, installers, config blobs, or secret-bearing historical material.
- Lifeline remains out of scope unless a later derivative explicitly becomes execution or runtime control logic.
- `_stack` remains out of scope unless a later derivative explicitly becomes stack automation or workflow-operator behavior.
- ATLAS root remains projection-only for this seam: doctrine, registration posture, and cross-repo visibility.

## Initial Pattern Payload

The first derivative pack is intentionally narrow:

- deterministic-first behavior over unsupported guesswork
- explicit provenance and confidence before reuse
- bounded convergence through seams instead of filesystem consolidation
- portable path discipline instead of machine-local assumptions
- owner-repo truth stays in the owner repo

These are candidate reusable Playbook principles, not proof that all Verta historical ideas should be promoted.

## Deterministic Promotion Criteria

The derivative pack is promotion-ready only when every criterion below is satisfied:

1. Every admitted pattern is rewritten as Playbook-owned language rather than copied raw Verta text.
2. Every admitted pattern links back to reviewed provenance surfaces.
3. No admitted section depends on secret-bearing, executable, or private operational material.
4. The owner repo remains explicit as `playbook`.
5. The verification command is recorded and reproducible.
6. ATLAS root projection docs reflect the same seam boundary and do not widen scope.
7. Any request to move beyond pattern doctrine into runtime behavior is routed through a new owner-seam decision.

## Downstream Routing Guidance

| Derivative type | Default route | Boundary rule |

| stack automation, orchestration, worker coordination | `_stack` | allowed only after stack-automation scope is explicit and separately reviewed |

- provenance references are explicit and reviewable

- the derivative pack has admitted, rejected, and pending review state recorded explicitly
- deterministic promotion criteria are documented
- Playbook verification and docs audit pass

## Rule

Verta-Core derivatives can be promoted only after they are rewritten as owner-repo artifacts with explicit provenance, trust boundary, and verification evidence.

## Pattern

The first safe absorption surface is a Playbook pattern pack because governance and architecture knowledge can be reviewed without granting runtime authority.

## Failure Mode

Jumping from Verta-derived ideas directly into Lifeline, `_stack`, or app runtime behavior skips the proof layer and risks converting untrusted historical material into executable stack behavior.
